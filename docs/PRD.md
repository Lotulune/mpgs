# MPGS MVP 产品需求文档

> 本文是 `0.1` / M0–M6 的已确认基线。下一阶段账号、社区想玩、头像、AI 用户配置与真实内容闭环见 [MPGS 0.2 产品需求文档](PRD_0.2.md)。

| 字段 | 内容 |
| --- | --- |
| 产品版本 | MVP `0.1` |
| 文档状态 | 已确认设计基线，允许进入开发 |
| 基线日期 | 2026-07-13 |
| 目标用户 | 固定或半固定熟人游戏小组 |

## 1. 背景

Steam 的热门榜和通用推荐会把在线人数高、竞技性强、依赖公共匹配的游戏排在前面，但这不等同于适合熟人联机。用户需要发现以下游戏：

- 能创建私人房间、P2P 房间或专用服务器。
- 适合固定的 2～8 人小组，而不是依赖大量陌生玩家维持体验。
- 即使当前在线人数较少，只要能自建服务器或私人联机，仍然值得游玩。
- 新作尚未积累大量评论时，可结合 Demo、联机形态和早期评价进行审慎推荐。

## 2. 产品目标

MVP 必须证明三件事：

1. 能稳定建立和更新一个可信的 Steam 多人游戏候选库。
2. 默认排序确实更偏向熟人联机，而不是热门网游榜。
3. AI 能理解自然语言需求、补充分析和解释，但不会创造不存在的游戏或覆盖硬性约束。

## 3. 非目标

以下能力不属于 MVP：

- Steam 登录、好友关系和多人游戏库交集。
- Android/iOS 应用、推送通知和社交组队。
- 用户发布评论、社区帖子或开放式内容平台。
- 自动购买、代购、交易、激活码或价格预测。
- 多主数据库、active-active 服务端和跨节点共享 SQLite。
- 基于大规模用户行为训练 Learning-to-Rank 模型。

## 4. 用户画像

### 4.1 固定游戏小组组织者

通常负责寻找下一款游戏，需要知道人数、开服方式、学习成本、价格和能否长期游玩。

### 4.2 轻度合作玩家

不喜欢高压竞技或陌生人匹配，希望快速建房、随进随出，每次游玩约 30～120 分钟。

### 4.3 生存建造小组

愿意自建服务器，关注世界持久化、最大人数、服务器平台、模组和停服后可玩性。

## 5. 核心概念

- **最近发售**：正式发售不超过 180 天的游戏。
- **即将发售/Demo**：未来有发售计划、Coming Soon、已有 Demo 或公开 Playtest 的游戏。
- **人气老游**：发售超过 180 天，当前仍有显著活跃度；允许降低口碑权重，但保留质量底线。
- **老牌联机**：发售超过 180 天且未进入人气老游的其余多人候选；按评价、熟人适配和数据可信度排序，不与人气老游重复。
- **熟人联机适配度**：私人房间、自建服、合作性、人数匹配、公共玩家依赖程度等特征的组合评分。
- **证据**：支持某个特征或推荐理由的来源记录，包含来源类型、时间和可信度。

## 6. MVP 范围

### 6.1 P0 用户功能

| 编号 | 功能 | 要求 |
| --- | --- | --- |
| FR-001 | 匿名初始化 | 首次启动生成设备级匿名标识；不登录也能完整浏览 |
| FR-002 | 偏好设置 | 常用人数、合作/竞技、预算、平台、单次时长、自建服意愿、语言 |
| FR-003 | 四类推荐流 | 每个条目显示核心理由、风险提示、证据时间和数据新鲜度 |
| FR-004 | 发售日历 | 按日期查看即将发售游戏，可只看已有 Demo/Playtest 的游戏 |
| FR-005 | 搜索与过滤 | 名称/语义搜索；人数、模式、平台、价格、Demo、发售状态过滤 |
| FR-006 | 游戏详情 | 展示联机方式、人数、评价、CCU 趋势、价格、平台、推荐理由和来源 |
| FR-007 | AI 推荐 | 接受自然语言需求，只能在检索候选集中推荐，并返回结构化理由和风险 |
| FR-008 | 反馈 | 喜欢、不感兴趣、玩过、人数不合适、太竞技、开服麻烦 |
| FR-009 | 离线缓存 | 断网时可浏览上次成功同步的推荐流和详情 |
| FR-010 | 跳转 Steam | 使用标准 Steam 商店链接打开对应 AppID |

### 6.2 P0 内部功能

| 编号 | 功能 | 要求 |
| --- | --- | --- |
| FR-101 | 数据源运行状态 | 查看上次成功时间、错误、限流和待重试数量 |
| FR-102 | 特征证据 | 查看某项联机特征来自官方数据、推断还是人工确认 |
| FR-103 | 人工校正 | 覆盖错误人数、联机方式、服务状态和错误 App 关系 |
| FR-104 | 推荐调试 | 输入用户偏好后查看候选过滤、分项得分、AI 调整和淘汰原因 |
| FR-105 | 算法配置 | 版本化管理阈值与权重；发布前可运行黄金测试集 |

### 6.3 P1 功能

- 新游/Demo 本地提醒。
- 推荐条目分享链接。
- 价格区间和折扣过滤增强。
- 管理界面的批量审核与导入导出。

P1 不得阻塞 MVP 发布。

## 7. 核心用户流程

### 7.1 首次使用

1. 用户选择常用人数、合作/竞技偏好、预算和单次时长。
2. 客户端请求四类推荐流。
3. 用户查看明确理由，例如“四人合作、私人房间、不依赖公共匹配”。
4. 用户通过反馈修正偏好，客户端刷新轻量个性化排序。

### 7.2 自然语言推荐

1. 用户输入“3 个人，晚上玩一小时，不要太卷，最好能反复刷”。
2. 服务端把需求转换为结构化过滤条件。
3. SQL、全文和向量检索生成候选；确定性推荐器排序。
4. AI 只分析 Top 20，并输出 3～10 个结果、风险和证据引用。
5. AI 超时、限额或输出无效时，立即返回确定性推荐结果。

### 7.3 查看新游

1. 用户进入新游页，在“最近发售”和“即将发售/Demo”间切换。
2. 无足够评论的新游显示“早期数据”标识，不伪装成高置信推荐。
3. 发售日期不确定或发生变更时显示日期可信度与最近更新时间。

## 8. 体验原则

- 首屏是可使用的推荐流，不是营销页。
- 每个推荐都回答“为什么适合这个小组”和“可能不适合在哪里”。
- 数据未知时显示未知，不把 Steam 标签或 AI 推断包装成确定事实。
- 人气只是体验健康指标之一，不应压过私人联机和自建服能力。
- 默认设置下，合作/自建服代表游戏应优先于公共匹配核心游戏。

## 9. 非功能要求

| 编号 | 领域 | MVP 要求 |
| --- | --- | --- |
| NFR-001 | 普通 API 性能 | 缓存命中时服务端 P95 小于 500 ms，不含公网传输 |
| NFR-002 | AI 延迟 | AI 请求服务端超时上限 12 s；超时返回非 AI 结果 |
| NFR-003 | 可用性 | Steam 或 AI 数据源故障时仍能读取最近一次成功快照 |
| NFR-004 | 数据新鲜度 | UI 显示更新时间；过期数据按字段类型降置信度 |
| NFR-005 | 安全 | Steam/AI Key 只在服务端；客户端无通用 SQL 或任意内部工具能力 |
| NFR-006 | 隐私 | 匿名可用；遥测可关闭；不采集 Steam 密码和非必要个人数据 |
| NFR-007 | 可解释 | 返回算法版本、主要分项和证据 ID；AI 不得引用未检索数据 |
| NFR-008 | 可移植 | 服务端构建覆盖 Windows/Linux 的 x86_64、aarch64 |
| NFR-009 | 可恢复 | SQLite 使用在线一致性备份；恢复流程有自动化验证 |
| NFR-010 | 成本 | AI 有请求上限、缓存和单次候选数量限制，可按功能关闭 |

## 10. 数据规模与上线门槛

数据可行性门槛：

- 至少建立 2,000 个多人候选游戏的规范化记录。
- 至少 300 个重点候选具备可信度不低于 `0.70` 的熟人联机特征。
- 四个分区均能生成至少 20 个有效候选；若某日新游不足，允许如实少于 20 个。
- 重点候选评价与 CCU 快照能完成连续 7 天采集，以验证趋势计算。

这些是 MVP 验收门槛，不代表需要抓取 Steam 全量详情。

## 11. 推荐质量验收

- 建立至少 50 个游戏的人工标注集，包含合作、自建服、竞技匹配、MMO、停服和人数不匹配案例。
- 默认偏好下，帕鲁、方舟、深岩银河、雨中冒险 2 的熟人适配排序应高于 CS2、永劫无间。
- 当“偏好竞技”显著提高时，竞技游戏允许上升，但仍需显示公共匹配依赖。
- AI 返回的所有 AppID 必须来自本次候选集，自动化测试通过率 `100%`。
- 游戏硬条件不满足时，AI 不得通过调高分数恢复该候选。

## 12. 产品指标

MVP 不以点击率作为唯一指标，至少记录：

- 推荐详情打开率、Steam 商店跳转率、Demo 跳转率。
- 喜欢率、不感兴趣率及原因分布。
- 推荐覆盖率、类型/发行商多样性、重复率。
- AI 有效响应率、回退率、平均成本和无效 AppID 拦截数。
- 数据源新鲜度、特征未知率和人工校正率。

所有行为指标都必须遵循隐私设置，并与算法版本关联。

## 13. MVP 发布验收

满足以下条件才可标记 `0.1.0`：

1. Windows x86_64 客户端完成主要用户流程；macOS 和 Linux 至少完成打包冒烟测试。
2. 四类推荐、搜索、详情、偏好、反馈和 AI 推荐均可用。
3. AI、Steam API 或网络不可用时，离线缓存和确定性推荐仍可使用。
4. 数据质量、黄金推荐集、API 契约、迁移、备份恢复和安全测试通过。
5. 没有 API Key、个人数据或任意 SQL 能力进入客户端包和日志。
6. 发布物具备版本号、算法版本和数据快照时间；签名策略有明确记录。

## 14. 已确认决策

- 使用单主 SQLite，数据库与应用服务同机。
- 使用 Tauri 2 桌面客户端，前端采用 React/TypeScript，核心服务与算法采用 Rust。
- MVP 匿名可用，Steam 账号与小组游戏库交集延后。
- 确定性算法是主推荐器，AI 只做检索理解、有限二次排序和解释。
- AI Provider、向量检索和存储通过 Rust 接口隔离，避免绑定单一供应商。
