# MPGS Public Discovery Service Migration Roadmap

- Document date: 2026-06-07
- Scope: migrating the current local Tauri application into a lightweight user client plus a self-hostable public discovery service

## Confirmed Direction

MPGS will move from a local all-in-one Tauri application to a split architecture:

- The user client connects to a configured service address.
- The public discovery service owns the public game library, Steam/LLM integration, discovery jobs, AI analysis, and management operations.
- Personal game state remains local to the user client.
- The management UI is a separate frontend entry from the user client.

## Target Architecture

```text
apps/
  client-ui/              # Tauri user client UI
  admin-ui/               # Management UI served by the service

crates/
  mpgs-core/              # Shared Rust domain logic, scoring, Steam/LLM adapters
  mpgs-server/            # Axum REST API, Postgres, OpenAPI, admin auth, workers

src-tauri/                # Lightweight Tauri shell
```

## Service Responsibilities

The public discovery service owns:

- public game library data
- Steam metadata sync and discovery
- AI analysis and batch processing
- Postgres-backed job queues
- management configuration state
- first-run setup flow
- management audit logs
- managed restart through service self-exit and Docker Compose restart policy

The service may start with an empty public library. Existing local SQLite data is not required for the new service; the public library is generated by server-side discovery after first-run configuration.

The first official self-hosting target is Docker Compose. TLS reverse proxying is not mandatory inside the default Compose stack, but the project should provide a Caddy profile as the official HTTPS example.

The service does not own:

- a user's personal favorite/wishlist/follow/history state
- client-local read cache
- client service-address storage

## Client Responsibilities

The user client owns:

- service address storage
- service identity verification
- local personal game state
- public library read cache for weak offline browsing
- external link opening

The user client does not own:

- Steam API keys
- LLM API keys
- discovery jobs
- public library writes
- AI analysis jobs

## Migration Phases

### Workspace Strategy

Rust workspace migration should happen before frontend workspace migration:

- Add a root Cargo workspace for `crates/mpgs-core`, `crates/mpgs-server`, and `src-tauri`.
- Keep the frontend as one npm package during the first migration slices.
- Split frontend source by entry responsibility first, such as `src/client`, `src/admin`, and `src/shared`.
- Only introduce npm workspaces later if the admin UI and client UI need independent package boundaries.

### Phase 1: Extract Core

- Create `mpgs-core`.
- Move shared models, scoring, recommendation, Steam mapping, LLM adapters, and analysis logic out of Tauri command modules.
- Keep current Tauri behavior working while the shared crate emerges.
- First extract only pure domain and mapping logic:
  - models
  - scoring
  - recommendation
  - multiplayer detection rules
  - pure game-analysis rule reports
  - Steam response mapping helpers
- Do not extract Tauri commands, SQLite persistence, runtime state, or background task orchestration in the first slice.

### Phase 2: Build Server

- Create `mpgs-server` with Axum.
- Add `/api/v1/service-info`.
- Add public read APIs for dashboard summary, paginated games, game detail, and analysis.
- Add Postgres schema split between public catalog data and ops data.
- Add explicit database migrations and versioned service configuration files.
- Add OpenAPI generation from Rust types.

### Phase 3: Add Management Surface

- Create `admin-ui` as a separate frontend entry.
- Add setup-token first-run flow.
- Add admin-token login with short-lived HttpOnly session cookies.
- Add server-side key configuration writing outside Postgres.
- Add task controls, failures, status views, and audit log views.

### Phase 4: Add Managed Restart

- Add service self-restart after validating pending configuration.
- Require Docker Compose restart policy for managed restart.
- Ensure the main service never controls Docker, mounts the Docker socket, or invokes arbitrary host commands.

### Phase 5: Slim the Tauri Client

- Replace Tauri `invoke` calls for public data with REST API calls.
- Keep only local service settings, personal game state, and read cache in client storage.
- Add service-address verification and HTTPS/LAN rules.

### Phase 6: Cleanup and Documentation

- Remove or freeze old client-side discovery, sync, and AI command paths.
- Update README for self-hosted server deployment and client connection.
- Document backup, key rotation, restart policy, HTTPS setup, and OpenAPI generation.

## First Implementation Slice

The safest first slice is:

1. Create `mpgs-core`.
2. Move non-Tauri scoring and model code into it.
3. Stand up `mpgs-server` with `/api/v1/service-info`.
4. Add generated OpenAPI for `service-info`.
5. Add a client-side service-address validation screen using that endpoint.

This proves the service boundary without migrating the entire game library or task system at once.

## First Release Artifacts

The first split-architecture release should produce:

- Windows Tauri user client installer
- `mpgs-server` Docker image
- Docker Compose deployment files
- optional Caddy Compose profile or override
- example service configuration files
- generated OpenAPI JSON

It should not produce a restart-helper image.
